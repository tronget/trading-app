# Скриншот-тестирование UI

Экраны Android-приложения (`#1`) проверяются **скриншот-тестами** на связке
**Roborazzi + Robolectric** — без эмулятора и без реального устройства. Это и
способ ловить визуальные баги, и (бонусом) источник картинок для отчёта.

## Что такое скриншот-тест

Обычный тест проверяет логику: «функция вернула 42». Но как проверить, что
**экран не разъехался**? Что кнопка на месте, цвета верные, текст не наслоился?

**Скриншот-тест** (snapshot testing) делает так:

1. рендерит экран (`@Composable`) в картинку;
2. сравнивает её с **эталонной** («golden image»), сохранённой ранее;
3. отличия → тест падает и показывает diff (что было / что стало / разница).

Эталоны коммитятся в репозиторий. Изменил UI осознанно — **перезаписал** эталоны
(`recordRoborazziDebug`). Сломал случайно — тест поймает на CI ещё до ревью.

> [!tip] Зачем это
> Визуальные регрессии (съехавший отступ, пропавшая кнопка, поломанный цвет)
> почти невозможно отловить логическими тестами и легко проворонить глазами.
> Скриншот-тест автоматизирует «а выглядит ли оно по-прежнему правильно».

## Robolectric: Android прямо на JVM

Обычные Android-тесты требуют устройство/эмулятор — а это медленно, и (в этом
проекте) [[#Почему именно без эмулятора|эмулятор вообще не запускается]].
**Robolectric** решает это: он реализует Android SDK **прямо на JVM**, так что
Android-код выполняется как обычный юнит-тест — быстро, в CI, без устройства.

Раньше Robolectric не умел настоящую **графику** (рисование пикселей). Для
скриншотов нужен реальный рендер, поэтому включён **Robolectric Native Graphics
(RNG)** — он рисует Compose в настоящий bitmap, который и сравнивается с эталоном.

## Roborazzi: сам движок скриншотов

**Roborazzi** — библиотека, которая поверх Robolectric делает снимок экрана,
сравнивает с эталоном и генерирует diff-картинки. Рабочий цикл:

```bash
# записать/обновить эталоны (когда UI изменён осознанно)
./gradlew recordRoborazziDebug

# проверить, что ничего не разъехалось (на CI)
./gradlew verifyRoborazziDebug
```

> [!note] Бонус: те же скриншоты — в отчёте
> Картинки из `app/screenshots/` (рынок, инструмент, портфель, история) — это и
> есть эталоны Roborazzi. Они же вставлены в отчёт по проекту (`report/`) и в
> презентацию (`presentation/`). Один артефакт — две пользы.

## Почему именно без эмулятора

Здесь причина не только в скорости. На машине, где собирался проект, **Android-
эмулятор стабильно падает** (segfault QEMU, регрессия в свежем ядре Linux) — гонять
UI-тесты на нём попросту нельзя. Robolectric + Roborazzi дают рендер экранов
**на JVM**, в обход эмулятора целиком. Хороший пример, как ограничение окружения
определяет выбор инструмента — ровно как [[Геометрическое броуновское движение|запрет
FPU в ядре]] продиктовал целочисленную математику в драйвере.
