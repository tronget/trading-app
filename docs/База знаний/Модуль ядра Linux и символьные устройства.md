# Модуль ядра Linux и символьные устройства

Компонент `#7` (`stock_gen_driver/stock_gen.c`) — это не обычная программа, а
**модуль ядра Linux**. Он живёт внутри ОС и создаёт «фальшивые» устройства
`/dev/stock0`…`/dev/stock3`, из которых можно читать котировки так же, как из
обычного файла. Звучит экзотично, но идея простая — давайте по порядку.

## Что такое модуль ядра

Ядро — это та часть ОС, которая рулит железом, памятью и процессами. Обычные
программы (наши Go/Kotlin-сервисы) работают в **user space** и не имеют прямого
доступа к железу — они просят ядро через системные вызовы (`read`, `write`…).

Модуль ядра — это код, который **подгружается прямо в ядро** на лету:

```bash
make                 # собрать stock_gen.ko
sudo insmod stock_gen.ko   # загрузить в ядро
lsmod | grep stock        # проверить
sudo rmmod stock_gen      # выгрузить
```

После загрузки модуль становится частью ядра — поэтому у него нет «main», а
есть `module_init()` / `module_exit()` (точки входа и выхода), и поэтому ошибка
в нём может уронить всю систему, а не один процесс. Отсюда же все странности
кода: нельзя пользоваться обычной libc, нельзя использовать числа с плавающей
точкой (FPU в ядре под запретом — см. [[Геометрическое броуновское движение]]),
а память берётся через `kzalloc`, а не `malloc`.

> [!question] Зачем вообще писать драйвер ради котировок?
> Это учебная задача: показать «источник данных на максимально низком
> уровне». В реальности котировки приходят по сети с биржи. Здесь же мы
> **имитируем железный тикер** — устройство, которое само, по таймеру, выдаёт
> новые цены.

## Символьное (character) устройство

В Linux устройства бывают двух типов:

- **block** — читаются блоками с произвольным доступом (диски);
- **character** — читаются **потоком байт по порядку** (клавиатура, `/dev/random`,
  последовательный порт).

Наш тикер — типичное character-устройство: новые цены идут потоком, «перемотать
назад» нельзя. Драйвер регистрирует набор операций (`file_operations`), и ядро
вызывает их, когда кто-то открывает/читает файл `/dev/stockN`:

```c
static const struct file_operations stock_fops = {
    .open    = stock_open,
    .read    = stock_read,     // отдать тик(и)
    .poll    = stock_poll,     // для epoll/select
    .unlocked_ioctl = stock_ioctl,  // настройка параметров
    .release = stock_release,
};
```

Благодаря этому **из user space всё выглядит как обычный файл** — на стороне Go
просто открывают `/dev/stock0` и читают 32 байта за тик. Это и есть философия
Unix «всё есть файл».

## Три интересных механизма внутри

**1. Блокирующее чтение + wait queue.** Когда читатель просит тик, а нового ещё
нет, поток не крутит CPU в пустую, а **засыпает** на `wait_event_interruptible`.
Таймер сгенерировал тик → `wake_up_interruptible` будит читателя. Это
«правильный» способ ждать в ядре.

**2. `poll`.** Реализация `poll` позволяет ждать сразу несколько устройств через
`epoll`/`select` — то самое «дождись, когда хоть где-то появятся данные».

**3. `ioctl` — боковой канал настройки.** Через `read` идут только данные. А как
поменять волатильность или интервал тика? Для этого есть `ioctl` —
«командный» вызов на том же файловом дескрипторе:

```c
#define STOCK_IOC_SET_PARAMS  _IOW('S', 1, struct stock_params)  // задать
#define STOCK_IOC_GET_PARAMS  _IOR('S', 2, struct stock_params)  // прочитать
#define STOCK_IOC_RESET       _IO ('S', 3)                       // сбросить
```

Можно задать базовую цену, волатильность и дрифт (в [базисных пунктах](https://ru.wikipedia.org/wiki/Базисный_пункт),
1 б.п. = 0,01 %) и интервал тика (по умолчанию 500 мс).

## Кольцевой буфер и «теряем старое»

Тики генерирует `hrtimer` (высокоточный таймер ядра) и складывает в **кольцевой
буфер** на 64 элемента. Если читатель тормозит и буфер переполняется — драйвер
**выкидывает самый старый тик**, а не блокирует генерацию. Это сквозной принцип
всей системы: лучше потерять устаревшую цену, чем затормозить всех (то же самое
делает [[WebSocket и fan-out котировок|fan-out в Gateway]] и
[[Go — горутины, каналы и backpressure|каналы в quotes-go]]).

## Формат записи (32 байта)

Каждый тик — ровно 32 байта, [little-endian](https://ru.wikipedia.org/wiki/Порядок_байтов):

| Смещение | Тип | Поле | Что это |
|---|---|---|---|
| 0–7 | u64 | `seq` | номер тика по порядку |
| 8–15 | s64 | `timestamp_ns` | время генерации, нс |
| 16–23 | s64 | `price_udollar` | цена в **микродолларах** (см. ниже) |
| 24–27 | u32 | `volume` | объём |
| 28–31 | u32 | `dev_idx` | номер устройства (→ тикер) |

Цена хранится не дробным числом, а целым числом микродолларов
($10,50 → `10_500_000`). Почему так — в заметке
[[Деньги, NUMERIC и числа с плавающей точкой]].
