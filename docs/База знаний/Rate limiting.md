# Rate limiting

**Rate limiting** — ограничение частоты запросов. Gateway (`#3`) не даёт одному
источнику долбить себя без меры: больше N запросов в секунду — получи `429 Too
Many Requests`. Простая на вид, но важная для выживания штука.

## Зачем это нужно

- **Защита от перегрузки.** Один кривой клиент в цикле, retry-шторм или просто
  всплеск трафика не должны положить сервис для всех остальных.
- **Защита от абьюза.** Перебор паролей на `/auth/login`, парсинг данных пачками —
  всё это упирается в лимит.
- **Справедливость.** Ресурсы делятся между клиентами, один не съедает все.

Это часть «защиты от перегрузки» наряду с
[[WebSocket и fan-out котировок|DROP_OLDEST в fan-out]] и
[[Голый SQL, JDBC и пул соединений|лимитом пула соединений к БД]] — у каждого
слоя свой предохранитель.

## Token bucket — как считают лимит

Самый ходовой алгоритм — **token bucket** («ведро жетонов»), его и использует
Ktor:

- есть ведро на N жетонов;
- каждый запрос съедает жетон;
- жетоны **доливаются** с постоянной скоростью (refill);
- ведро пустое → запрос отбивается (`429`).

Красота в том, что это разрешает **короткие всплески** (пока в ведре есть запас),
но держит **среднюю** скорость. Жёсткий счётчик «ровно N в секунду» так не умеет.

```kotlin
install(RateLimit) {
    register(RateLimitName("api")) {
        rateLimiter(limit = config.rateLimitPerSecond, refillPeriod = 1.seconds)
        requestKey { call -> call.request.origin.remoteHost }   // ключ = IP
    }
}
```

## Лимит на каждый IP отдельно

Ключевая деталь — **`requestKey = remoteHost`**: ведро **своё на каждый IP**, а
не одно общее. Иначе один шумный клиент исчерпал бы общий лимит и заблокировал
всех. Так у каждого клиента свой счётчик.

Когда лимит превышен, Ktor возвращает `429`, а
[[Ktor и корутины Kotlin|StatusPages]] оборачивает это в аккуратный JSON:

```kotlin
status(HttpStatusCode.TooManyRequests) { call, status ->
    call.respond(status, ErrorResponse("rate limit exceeded"))
}
```

> [!warning] Тонкости «на проде»
> - Лимит **на инстанс**. Если поднять несколько Gateway за балансировщиком,
>   каждый считает своё ведро — для общего лимита счётчик выносят в Redis.
> - **`remoteHost` за прокси/балансировщиком** — это IP прокси, а не клиента.
>   Тогда IP берут из заголовка `X-Forwarded-For` (и доверяют ему только от
>   своего балансировщика).
> Для учебного сценария «10 000 клиентов с одной машины» текущего варианта
> достаточно, но знать про это полезно.

> [!note] Почему лимит не мешает нагрузочному тесту
> [[Нагрузочное тестирование и перцентили|Имитатор]] подбирает темп так, чтобы
> оставаться в разумных рамках; цель теста — пропускная способность и задержки,
> а не пробить rate limit. Поэтому доля ошибок там — сотые доли процента.
