# Ордера, позиции и P&L

Чтобы понимать торговую логику Data-сервиса (`#4`), нужен небольшой биржевой
словарь. Здесь — по-человечески про ордера, позиции, среднюю цену и прибыль.
Код, который всё это считает, — `OrderService.kt`.

## Ордер: заявка на сделку

**Ордер** (order) — это заявка «хочу купить/продать». У него есть:

- **side** — направление: `BUY` (купить) или `SELL` (продать);
- **symbol** — что торгуем (`SBER`, `BTC`…);
- **qty** — количество;
- **type** — тип (см. ниже);
- **status** — жизненный цикл: `NEW` → `FILLED` (исполнен) / `CANCELLED` /
  `REJECTED`.

### Market vs Limit — два типа ордеров

| | Market (рыночный) | Limit (лимитный) |
|---|---|---|
| Смысл | «купи **сейчас** по текущей цене» | «купи, **когда** цена станет ≤ X» |
| Когда исполняется | сразу | когда рынок «дойдёт» до цены |
| Цена | какая есть | не хуже заданной |

В проекте:

- **Market** исполняется немедленно по последней цене из
  [[Redis — Pub-Sub и кэш|Redis-кэша]]. Нет цены в кэше → ордер `REJECTED`.
- **Limit** сохраняется со статусом `NEW` и ждёт. Фоновый
  [[PostgreSQL — транзакции, блокировки и матчинг|MatchingWorker]] на каждый тик
  проверяет: не пора ли? Логика «пересечения»:

```kotlin
fun limitCrossed(side, limitPrice, tickPrice) = when (side) {
    Side.BUY  -> tickPrice <= limitPrice   // покупатель ждёт цену пониже
    Side.SELL -> tickPrice >= limitPrice   // продавец ждёт цену повыше
}
```

> [!note] Это упрощённый матчинг «против рынка»
> Настоящая биржа сводит заявки покупателей и продавцов друг с другом (есть
> «стакан» заявок). Здесь же лимитка исполняется **против текущей рыночной цены**
> из потока тиков — этого достаточно для учебной модели. «Стакан» и матчинг
> клиент-против-клиента указаны как направление развития в отчёте.

## Позиция и средняя цена

**Позиция** (position) — сколько у тебя сейчас бумаг и **по какой средней цене**
ты их набрал. Средняя цена нужна, чтобы понимать прибыль.

Купил 10 акций по $100, потом ещё 10 по $120. Сколько «стоила» одна в среднем?

```
средняя = (10×100 + 10×120) / 20 = $110
```

Именно это считает `newAveragePrice` при каждой докупке (на `BigDecimal`, см.
[[Деньги, NUMERIC и числа с плавающей точкой]]). При продаже средняя **не
меняется** — просто уменьшается количество (мы фиксируем результат, а цена
оставшихся бумаг та же).

## P&L — прибыль и убыток

**P&L** (Profit and Loss) — сколько ты заработал или потерял. «Бумажный»
(нереализованный) P&L по позиции:

```
P&L = (текущая цена − средняя цена) × количество
```

Купил по $110, сейчас $115, держишь 20 штук → P&L = (115−110)×20 = +$100.
Приложение показывает это на экране «Портфель», подсвечивая плюс зелёным, минус
красным. Стоимость портфеля = кэш + сумма позиций по текущим ценам.

## Идемпотентность: `clientOrderId`

Сеть может оборвать ответ, и клиент повторит запрос. Чтобы **не создать два
ордера из одного**, клиент присылает `clientOrderId` — свой уникальный
идентификатор заявки. Сервер перед вставкой проверяет:

```kotlin
request.clientOrderId?.let { id ->
    OrderRepo.findByClientOrderId(c, id)?.let { return@tx it }  // уже есть — вернуть его
}
```

Повторный запрос с тем же id вернёт **существующий** ордер, а не создаст дубль.
Это и есть **идемпотентность** — «повтори сколько угодно раз, результат один».
На уровне БД её страхует ограничение `client_order_id UNIQUE` (см.
[[PostgreSQL — транзакции, блокировки и матчинг]]).

## Всё это — в одной транзакции

Любое исполнение (`executeFill`) меняет **три вещи разом**: пишет сделку (trade),
двигает позицию, двигает баланс. Всё — в одной
[[PostgreSQL — транзакции, блокировки и матчинг|транзакции]] с блокировкой счёта,
чтобы не было гонок и полусостояний. При нехватке денег/бумаг — `REJECTED`,
ничего не списано.
