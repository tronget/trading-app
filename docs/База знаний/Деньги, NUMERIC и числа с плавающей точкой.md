# Деньги, NUMERIC и числа с плавающей точкой

Сквозная тема всего проекта: **деньги нельзя хранить и считать в `float`/`double`**.
Это касается драйвера, обеих БД, JSON и Android. Давайте разберём, почему это
не паранойя, а реальный источник багов, и как проект это обходит.

## В чём проблема с `double`

Числа с плавающей точкой (`float`, `double`) — **двоичные**. А многие десятичные
дроби в двоичной системе представляются **бесконечной** дробью, как `1/3` в
десятичной. Классика:

```
0.1 + 0.2 == 0.30000000000000004   // не равно 0.3 !
```

Для денег это катастрофа: копейки «уплывают», суммы не сходятся, сравнения
ломаются, при тысячах операций ошибка накапливается. Никогда не считай деньги в
`double`.

> [!example] Почему это бьёт именно по бирже
> Тут складываются и умножаются цены × количества миллионы раз (тики, сделки,
> переоценка портфеля). Даже крошечная ошибка на операцию, помноженная на объём,
> превращается в расхождение баланса. Аудит такого не простит.

## Решение 1: фиксированная точка (целые числа)

Идея — хранить деньги в **целых числах** мелких единиц, а не в дробях. Целые
складываются и сравниваются точно.

В [[Модуль ядра Linux и символьные устройства|драйвере]] цена — это **микродоллары**:
целое число = доллары × 1 000 000. $10,50 → `10_500_000`. Бонус: в ядре всё
равно [[Геометрическое броуновское движение|нельзя использовать FPU]], так что
целые — единственный вариант.

## Решение 2: NUMERIC / Decimal (точная десятичная арифметика)

В БД и бизнес-логике используют **десятичный** тип с фиксированной точностью:

- PostgreSQL — `NUMERIC(18, 6)` (18 значащих цифр, 6 после запятой) для
  количеств и цен, `NUMERIC(18, 2)` для денежного баланса;
- [[ClickHouse и колоночные базы данных|ClickHouse]] — `Decimal(18, 6)` для цен;
- Kotlin — `BigDecimal` (а не `Double`!) во всей денежной логике.

`NUMERIC`/`BigDecimal` хранят число **точно в десятичном виде** — никаких
двоичных сюрпризов. Расплата — они медленнее `double`, но для денег корректность
важнее.

```kotlin
// средняя цена позиции — всё на BigDecimal, с явным округлением
fun newAveragePrice(oldQty, oldAvg, addQty, addPrice): BigDecimal =
    (oldQty * oldAvg + addQty * addPrice)
        .divide(totalQty, MathContext.DECIMAL64)
        .setScale(6, RoundingMode.HALF_UP)   // округление задаём явно
```

Важная деталь: при делении **округление задают явно** (`HALF_UP`, до 6 знаков) —
иначе `BigDecimal` кинет исключение на бесконечной дроби. Никакого «как-нибудь
округлится само».

## Решение 3: деньги в JSON — строками

Самое неочевидное. Когда сумма едет по сети в JSON, её **передают строкой**, а не
числом:

```json
{ "symbol": "SBER", "qty": "2.5", "price": "305.500000" }
```

Зачем `"2.5"`, а не `2.5`? Потому что JSON-число почти везде парсится в… `double`.
То есть стоит отдать `305.500000` числом — и принимающая сторона радостно
сконвертирует его в неточный `double`, обнулив все наши старания. Строка же
доедет байт-в-байт, и её распарсят прямо в `BigDecimal`. Поэтому в
[[02 — Контракты API|контракте API]] все money-поля — строки.

> [!tip] Правило большого пальца
> Деньги: храни и считай в `NUMERIC`/`BigDecimal` или целых мелких единицах;
> передавай строкой; округляй явно. `double` для денег — никогда.
