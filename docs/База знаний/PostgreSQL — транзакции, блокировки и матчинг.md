# PostgreSQL: транзакции, блокировки и матчинг

Деньги, счета, ордера, сделки и позиции живут в **PostgreSQL** (Data-сервис,
`#4`). Здесь важна не скорость аналитики, как в
[[ClickHouse и колоночные базы данных|ClickHouse]], а **корректность**: нельзя
списать деньги дважды, нельзя уйти в минус, нельзя потерять сделку. За это
отвечают транзакции и блокировки — самая «опасная» часть проекта, разберём
подробно.

## Транзакция и ACID

Транзакция — это группа операций, которая выполняется **целиком или никак**.
Купить акцию = записать сделку + уменьшить кэш + увеличить позицию. Если после
списания денег упадёт запись позиции — клиент потеряет деньги ни за что. Поэтому
все три действия идут в одной транзакции:

```kotlin
db.tx("place_order") { c ->        // BEGIN
    // ... записать сделку, изменить баланс и позицию ...
}                                  // COMMIT; при исключении — ROLLBACK
```

`db.tx` (см. [[Голый SQL, JDBC и пул соединений]]) ставит `autoCommit = false`,
выполняет блок и делает `commit()`, а при любом исключении — `rollback()`.
Это буква **A** (atomicity) из **ACID**. Буква **C** (consistency) подкреплена в
самой схеме:

```sql
cash_balance NUMERIC(18,2) NOT NULL CHECK (cash_balance >= 0)
```

`CHECK` — это **страховка на уровне БД**: даже если в коде проворонят проверку,
БД физически не даст балансу уйти в минус.

## Гонки и почему нужны блокировки

Представь: у клиента $100, и **два запроса на покупку по $80 приходят
одновременно**. Оба читают баланс ($100), оба видят «хватает», оба списывают —
итог −$60. Это **состояние гонки** (race condition).

Лечится **блокировкой строки** на время транзакции — `SELECT … FOR UPDATE`:

```sql
SELECT cash_balance FROM accounts
WHERE user_id = ? AND currency = ? FOR UPDATE   -- ← строка заблокирована
```

`FOR UPDATE` говорит: «я собираюсь менять эту строку, держите её только за мной
до конца транзакции». Второй запрос **подождёт** на этой строке, пока первый не
сделает commit, и уже потом увидит честный остаток. Гонки нет. В коде это
`AccountRepo.lockCash(...)` — первое, что делает исполнение ордера.

> [!note] Только параметризованные запросы
> Везде `?`-плейсхолдеры и `setString/setLong`, а не склейка строк. Это не
> только защита от SQL-инъекций — это требование, см.
> [[Голый SQL, JDBC и пул соединений|Голый SQL]].

## Матчинг лимиток: `FOR UPDATE SKIP LOCKED`

Самое интересное. Лимитные ордера (см. [[Ордера, позиции и P&L]]) исполняются не
сразу, а когда цена их «пересечёт». Этим занимается фоновый
`MatchingWorker`: он слушает поток тиков и на каждый тик ищет подходящие
лимитки:

```kotlin
prices.ticks.collect { tick ->
    db.tx("match_limit_orders") { c ->
        val crossed = OrderRepo.lockCrossedLimitOrders(c, tick.symbol, tickPrice)
        crossed.forEach { executeFill(c, it, tickPrice) }
    }
}
```

Внутри `lockCrossedLimitOrders` — `SELECT … FOR UPDATE SKIP LOCKED`. Зачем
`SKIP LOCKED`? Тики прилетают часто, и две итерации матчинга могут пересечься во
времени. Без `SKIP LOCKED` вторая встала бы ждать строки, которые уже
обрабатывает первая. С `SKIP LOCKED` она **просто пропускает занятые строки** и
берёт свободные — никто никого не ждёт, никаких дедлоков, ордер не исполнится
дважды.

> [!tip] `SKIP LOCKED` = «очередь задач» в SQL
> Это стандартный приём для конкурентной обработки: несколько воркеров разбирают
> строки-задачи из одной таблицы, и каждая строка достаётся ровно одному. Тот же
> паттерн используют самодельные очереди задач поверх PostgreSQL.

## Идемпотентность

Сеть ненадёжна: клиент отправил ордер, не дождался ответа, повторил. Чтобы не
создать два ордера, у каждого есть `client_order_id UNIQUE`, и перед вставкой
сервис проверяет — нет ли уже такого. Если есть — возвращает существующий ордер.
Подробнее в [[Ордера, позиции и P&L]].
