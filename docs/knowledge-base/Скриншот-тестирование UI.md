# Скриншот-тестирование UI

Экраны Android-приложения проверяются скриншот-тестами на базе **Roborazzi и
Robolectric**. Для запуска не нужен эмулятор или реальное устройство.

## Что такое скриншот-тест

Обычный тест проверяет данные и поведение. Скриншот-тест дополнительно проверяет
расположение элементов, цвета и отображение текста.

**Скриншот-тест** работает так:

1. рендерит экран (`@Composable`) в картинку;
2. сравнивает её с сохранённым эталоном;
3. при отличиях тест падает и сохраняет изображения до и после изменения.

Эталоны коммитятся в репозиторий. Изменил UI осознанно — **перезаписал** эталоны
(`recordRoborazziDebug`). Сломал случайно — тест поймает на CI ещё до ревью.

Так можно автоматически находить визуальные регрессии: неверный отступ,
пропавшую кнопку или изменившийся цвет.

## Robolectric: Android прямо на JVM

Обычные Android-тесты требуют устройство/эмулятор — а это медленно, и (в этом
проекте) [[#Почему именно без эмулятора|эмулятор вообще не запускается]].
**Robolectric** решает это: он реализует Android SDK **прямо на JVM**, так что
Android-код выполняется как обычный юнит-тест — быстро, в CI, без устройства.

Для скриншотов нужен настоящий рендер пикселей. Поэтому включён режим
`Robolectric Native Graphics`, который рисует Compose в изображение и сравнивает
его с эталоном.

## Roborazzi: сам движок скриншотов

**Roborazzi** — библиотека, которая поверх Robolectric делает снимок экрана,
сравнивает с эталоном и генерирует diff-картинки. Рабочий цикл:

```bash
# записать/обновить эталоны (когда UI изменён осознанно)
./gradlew recordRoborazziDebug

# проверить, что ничего не разъехалось (на CI)
./gradlew verifyRoborazziDebug
```

Эталоны из `app/screenshots/` также используются в отчёте и презентации.

## Почему именно без эмулятора

Здесь причина не только в скорости. На машине, где собирался проект, **Android-
эмулятор стабильно падает из-за ошибки QEMU, поэтому запускать на нём UI-тесты
нельзя. Robolectric и Roborazzi дают рендер экранов
**на JVM**, в обход эмулятора.
