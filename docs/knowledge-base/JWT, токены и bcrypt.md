# JWT, токены и bcrypt

Логин в системе устроен **без серверных сессий**: сервер не помнит, кто залогинен.
Вместо этого клиент носит с собой подписанный «пропуск» — **JWT**. А пароли в БД
лежат не как есть, а в виде **bcrypt-хэшей**. Разберём обе вещи.

## Зачем хэшировать пароль (bcrypt)

Пароли **никогда** не хранят в открытом виде: утечёт база — утекут все пароли (а
люди их переиспользуют). Хранят **хэш** — необратимый «отпечаток». При логине
хэшируют введённый пароль и сравнивают с сохранённым.

Но обычный хэш (SHA-256) для паролей **не годится** — он слишком быстрый.
Злоумышленник переберёт миллиарды вариантов в секунду на GPU. **bcrypt**
сделан специально «медленным» и с двумя свойствами:

- **соль** (salt) — случайная добавка к каждому паролю, поэтому одинаковые
  пароли дают разные хэши, и радужные таблицы бесполезны;
- **cost factor** — настраиваемая «цена» вычисления (в проекте — `bcryptCost`).
  Чем мощнее железо со временем — тем выше ставим cost. Подбор становится
  непрактично дорогим.

> [!tip] Правило
> Для паролей — только медленные «password hashing» функции (bcrypt, scrypt,
> argon2). SHA/MD5 для паролей — ошибка.

## JWT: пропуск с подписью

**JWT** (JSON Web Token) — строка из трёх частей через точку:

```
header.payload.signature
eyJhbGc… . eyJzdWI… . SflKxw…
```

- **header** — тип и алгоритм подписи;
- **payload** — «claims», полезные данные: кто это (`sub` = userId), `login`,
  тип токена, срок годности (`exp`);
- **signature** — подпись header+payload секретным ключом сервера.

Payload **не шифруется**, а только **подписывается** (он читается кем угодно через
base64 — секретов туда класть нельзя). Смысл подписи: клиент не может подделать
содержимое. Поменяет `userId` в payload — подпись не сойдётся, и сервер отвергнет
токен. Проект использует алгоритм **HS256** (HMAC-SHA256): одним секретом и
подписывают, и проверяют.

```kotlin
JWT.create()
   .withSubject(userId.toString())
   .withClaim("login", login)
   .withClaim("type", "access")
   .withExpiresAt(Instant.now().plus(accessTtlMinutes, MINUTES))
   .sign(Algorithm.HMAC256(secret))
```

> [!note] Stateless — и в этом сила и слабость
> Сервер проверяет подпись и читает `userId` **прямо из токена**, не ходя в БД и
> не храня сессий. Это масштабируется идеально: любой инстанс Gateway проверит
> токен сам (см. [[API Gateway и микросервисы]]). Обратная сторона: выданный
> токен нельзя «отозвать» досрочно — он валиден до `exp`. Поэтому…

## Access + refresh: два токена

…access-токен делают **короткоживущим** (15 минут). Утечёт — протухнет быстро.
Но не гонять же логин/пароль каждые 15 минут? Для этого есть второй,
**refresh-токен** (живёт 7 дней): по нему можно получить новый access, не вводя
пароль.

Различаются они claim'ом `type` (`access` / `refresh`), и проверяются **разными
верификаторами** — refresh-токеном нельзя ходить в API, а access нельзя обновлять:

```kotlin
val verifier = JWT.require(algorithm).withClaim("type", "access").build()
val refreshVerifier = JWT.require(algorithm).withClaim("type", "refresh").build()
```

Полный танец «протух access → меняем по refresh → повторяем запрос» происходит
**автоматически на клиенте** — см. [[DataStore и авто-refresh токена]].

## Где это в проекте

- Gateway выдаёт токены (`/auth/register`, `/auth/login`, `/auth/refresh`) и
  проверяет их Ktor-плагином `Authentication { jwt }`.
- На защищённых маршрутах `userId` берётся **из токена**, а не из тела запроса —
  поэтому один пользователь не сможет торговать от имени другого.
- WebSocket проверяет токен вручную (`verifyAccess`), потому что там плагина
  аутентификации нет — токен передаётся в query (`/ws?token=…`).
