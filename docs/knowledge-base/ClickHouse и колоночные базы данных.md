# ClickHouse и колоночные базы данных

ClickHouse используется для истории котировок. Это не база «для всего подряд»,
а отдельное хранилище для данных, которых много и по которым часто строятся
выборки по времени: график цены, свечи, максимум/минимум за период.

## Почему не хранить это только в PostgreSQL

PostgreSQL хорошо подходит для счетов, заявок и сделок. Там важны транзакции:
списать деньги, создать заявку, записать сделку. Всё должно пройти целиком или
откатиться.

С котировками другая ситуация. Тики постоянно добавляются. Их много, и чаще
всего читается не одна строка, а диапазон: например, цена `SBER` за час.
Для такой истории ClickHouse подходит лучше.

Главная идея колоночной базы простая: данные хранятся по столбцам. Если мне
нужна только цена и время, база не читает лишние поля. На больших объёмах это
заметно.

## Где ClickHouse уместен

Его стоит использовать не как замену PostgreSQL, а как отдельное хранилище под
аналитику и историю. В моём проекте это выглядит так:

- актуальные операции и состояние пользователя — в PostgreSQL;
- поток котировок и свечи для графика — в ClickHouse.

Так разделение получается логичным. PostgreSQL отвечает за корректность
операций, ClickHouse — за быстрое чтение истории.

## Техническая деталь

Для тиков используется таблица на `MergeTree`:

```sql
CREATE TABLE trading.ticks (
    symbol LowCardinality(String),
    ts     DateTime64(3),
    price  Decimal(18, 6),
    volume UInt32,
    seq    UInt64
) ENGINE = MergeTree ORDER BY (symbol, ts);
```

`ORDER BY (symbol, ts)` важен не только для сортировки. Данные физически
раскладываются так, чтобы быстро читать диапазон по инструменту и времени.

Свечи считаются через материализованное представление. Это удобно: часть работы
делается при записи тиков, а не каждый раз при открытии графика.
