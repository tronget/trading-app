# ClickHouse и колоночные базы данных

История котировок хранится в **ClickHouse**. Эта база хорошо подходит для
больших потоков данных и запросов по временным диапазонам.

## Строковые vs колоночные БД

PostgreSQL хранит данные **по строкам**: вся строка лежит рядом на диске. Это
удобно, когда работаешь с отдельными записями целиком: «дай заказ
№42», «обнови баланс пользователя». Это **OLTP** — много мелких транзакций.

Колоночная БД хранит данные **по столбцам**: все `price` лежат вместе, все `ts`
вместе, все `symbol` вместе. Идеально для **OLAP** — аналитики по огромным
таблицам: «средняя цена SBER за день», «максимум за час». Почему быстрее:

- читаешь с диска **только нужные столбцы**, а не все поля;
- одинаковые данные в столбце жмутся в разы лучше (миллион почти одинаковых
  цен → крошечный объём);
- процессор обрабатывает столбец пачкой (векторно).

Например, для расчёта средней цены ClickHouse читает только столбец `price`.
Счета и сделки остаются в PostgreSQL, потому что для них важнее транзакции и
точечные изменения строк.

## MergeTree — движок таблиц

В ClickHouse у каждой таблицы есть **движок**. Основной — `MergeTree`. Тики:

```sql
CREATE TABLE trading.ticks (
    symbol LowCardinality(String),
    ts     DateTime64(3),
    price  Decimal(18, 6),
    volume UInt32,
    seq    UInt64
) ENGINE = MergeTree ORDER BY (symbol, ts);
```

Что тут интересного:

- **`ORDER BY (symbol, ts)`** — это не просто сортировка, это **разреженный
  первичный индекс**. Данные физически лежат упорядоченными, поэтому запрос «SBER
  за последний час» прыгает сразу в нужный кусок, не сканируя всё.
- **`LowCardinality(String)`** — оптимизация для столбца с малым числом разных
  значений (у нас 4 тикера). ClickHouse хранит их как словарь чисел — экономия
  и скорость.
- **`Decimal(18, 6)`** — точные значения цены, без ошибок округления как у
  `float`.
- Название «Merge»Tree — потому что данные пишутся пачками-кусками, а в фоне
  движок **сливает** их в бóльшие. Поэтому ClickHouse обожает **батчевые
  вставки** и не любит вставки по одной строке — оттого quotes-go копит тики и
  пишет раз в секунду / по 5000 строк.

## Свечи: материализованное представление + AggregatingMergeTree

График рисуется свечами. Для каждой минуты нужны цена открытия, максимум,
минимум и цена закрытия. Считать их каждый раз из сырых тиков дорого. Поэтому
используется **материализованное представление**. Каждый раз, когда новые
тики падают в `ticks`, MV дописывает агрегаты в таблицу `candles_1m`:

```sql
CREATE MATERIALIZED VIEW trading.candles_1m_mv TO trading.candles_1m AS
SELECT symbol,
       toStartOfMinute(ts)    AS t,
       argMinState(price, ts) AS open,   -- цена самого раннего тика минуты
       max(price)             AS high,
       min(price)             AS low,
       argMaxState(price, ts) AS close,  -- цена самого позднего тика
       sum(toUInt64(volume))  AS volume
FROM trading.ticks GROUP BY symbol, t;
```

Таблица свечей на движке **`AggregatingMergeTree`** — она умеет «дослаивать»
частичные агрегаты. Тонкость: `open`/`close` хранятся как **состояния**
(`argMinState`/`argMaxState`), а читаются через `-Merge`-функции
(`argMinMerge`). Это потому, что свеча за минуту собирается из нескольких пачек
тиков, прилетевших в разное время, и движку нужно уметь их **досливать**, а не
перезаписывать.

Так свечи частично считаются во время записи. При открытии графика не нужно
снова обрабатывать все сырые тики.
