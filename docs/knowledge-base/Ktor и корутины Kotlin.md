# Ktor и корутины Kotlin

Gateway (`#3`) и Data-сервис (`#4`) написаны на **Kotlin + Ktor**. Ktor — это
лёгкий веб-фреймворк от создателей Kotlin, целиком построенный на **корутинах**.
Чтобы понять код сервисов, надо сначала понять корутины.

## Корутины: асинхронность, которая читается как обычный код

Сервер постоянно кого-то ждёт: ответа от БД, от Redis, от другого сервиса. Если
на каждый запрос вешать **поток ОС** и блокировать его на ожидании — потоки
кончатся уже на паре тысяч клиентов (каждый поток весит мегабайты).

Корутина — это «лёгкая задача», которую можно **приостановить** (suspend) на
ожидании, освободив поток для других задач, и **возобновить** позже. Тысячи
корутин крутятся на десятке потоков.

Магия в слове `suspend`:

```kotlin
suspend fun place(request: PlaceOrderRequest): OrderDto {
    val order = db.tx("place_order") { … }   // тут корутина «засыпает», ждя БД,
    return order                              // но поток не блокируется
}
```

Выглядит как обычный последовательный код — никаких колбэков и `.then().then()`.
Но в точке `db.tx(...)` корутина не держит поток, пока БД думает. Это
**асинхронность без callback hell** — главная фишка.

> [!note] Диспетчеры: где выполняется корутина
> JDBC — **блокирующий** API (см. [[Голый SQL, JDBC и пул соединений]]), он не
> умеет «засыпать» по-корутинному. Поэтому работа с БД обёрнута в
> `withContext(Dispatchers.IO)` — пул потоков, специально для блокирующих
> операций. Так блокирующий JDBC не застопорит потоки, обрабатывающие сеть.

## Structured concurrency

Корутины запускаются в **scope** и образуют дерево: родитель ждёт детей, а
отмена/ошибка распространяется по дереву. Нельзя «забыть» корутину — она привязана
к области видимости. Так
[[PostgreSQL — транзакции, блокировки и матчинг|MatchingWorker]] живёт фоновой
корутиной, пока работает приложение:

```kotlin
fun start(scope: CoroutineScope): Job = scope.launch {
    prices.ticks.collect { tick -> matchSymbol(...) }   // бесконечно слушает поток
}
```

## Ktor: сервер из плагинов

Приложение Ktor собирается как конструктор из **плагинов** (`install(...)`).
Вот реальный костяк Gateway:

```kotlin
install(ContentNegotiation) { json(...) }   // (де)сериализация JSON
install(WebSockets) { pingPeriod = 30.seconds }
install(RateLimit) { … }                     // см. [[Rate limiting]]
install(Authentication) { jwt("jwt") { … } } // см. [[JWT, токены и bcrypt]]
install(StatusPages) { … }                   // ошибки → аккуратный JSON

routing {
    get("/health") { call.respond(...) }
    rateLimit(RateLimitName("api")) {
        authRoutes(...); quoteRoutes(...)
        authenticate("jwt") { tradingRoutes(...) }  // только с валидным токеном
    }
}
```

Читается почти как оглавление: какие фичи включены и какие маршруты за чем
спрятаны. `StatusPages` — отдельно приятная штука: превращает исключения в
аккуратные ответы (`IllegalArgumentException → 422`, `429 → "rate limit
exceeded"`), чтобы наружу не лезли стектрейсы.

> [!tip] Почему Kotlin для бэкенда и клиента
> Один язык на сервере (Ktor) и в Android-приложении (см.
> [[Jetpack Compose, MVVM и StateFlow]]). Модели данных, стиль, корутины —
> общие; переключаться между клиентом и сервером проще.
