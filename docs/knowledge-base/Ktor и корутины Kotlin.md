# Ktor и корутины Kotlin

Gateway и Data-сервис написаны на Kotlin/Ktor. Я выбрал такой стек, потому что
он хорошо ложится на асинхронный сервер: много запросов, ожидание БД, Redis,
WebSocket и фоновые задачи.

## Почему здесь корутины

Сервер большую часть времени ждёт: ответ от БД, сообщение из Redis, сеть. Если
на каждое ожидание держать отдельный тяжёлый поток, при большом числе клиентов
это быстро станет проблемой.

Корутина легче. Она может приостановиться на ожидании и освободить поток для
другой работы. При этом код выглядит почти как обычный последовательный код,
без цепочек callback-функций.

Главный плюс такого подхода: асинхронность есть, но код остаётся читаемым как
обычная последовательность действий.

## Как Ktor устроен в проекте

Ktor собирается из плагинов. Отдельно подключаются JSON, WebSocket,
авторизация, лимиты и обработка ошибок. Это удобно: видно, какие возможности
есть у сервиса.

Gateway принимает внешние запросы, Data-сервис выполняет операции с данными, а
фоновые корутины слушают потоки котировок и запускают обработку.

## Важная оговорка

Не всё в проекте полностью неблокирующее. Работа с PostgreSQL всё равно
блокирующая, поэтому её надо выполнять отдельно от потоков, которые обслуживают
сетевые запросы. Иначе можно случайно занять серверные потоки ожиданием БД.

## Техническая деталь

Пример конфигурации Ktor:

```kotlin
install(ContentNegotiation) { json() }
install(WebSockets)
install(RateLimit)
install(Authentication) { jwt("jwt") { ... } }
install(StatusPages) { ... }

routing {
    get("/health") { call.respond(...) }
    authenticate("jwt") {
        tradingRoutes(...)
    }
}
```

Ошибки оборачиваются через `StatusPages`, чтобы клиент получал нормальный JSON,
а не стектрейс.
