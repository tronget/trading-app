# WebSocket и fan-out котировок

Gateway передаёт котировки клиентам по **WebSocket**. Внутри сервиса поток
раздаётся всем подключениям через fan-out.

## Почему WebSocket, а не обычный HTTP

Обычный HTTP — это «вопрос-ответ»: клиент спросил, сервер ответил, соединение
закрылось. Чтобы получать свежие цены, пришлось бы опрашивать сервер каждую
секунду. Это создаёт лишние запросы, задержку и расходует заряд телефона.

Варианты доставки «сервер → клиент»:

| Способ | Как работает | Минус |
|---|---|---|
| Polling | клиент дёргает `/quotes` по таймеру | задержка, трафик, батарея |
| SSE | один незакрытый HTTP, сервер шлёт строки | только в одну сторону |
| **WebSocket** | постоянное двустороннее соединение | чуть сложнее |

**WebSocket** — это апгрейд HTTP-соединения до постоянного двустороннего канала.
Открыли один раз — и дальше сервер **сам пушит** тики, как только они появились,
а клиент по тому же каналу шлёт команды (подписаться на `SBER`, отписаться).
Идеально для котировок и щадит батарею телефона (одно соединение вместо
бесконечного опроса).

```
client → { "action":"subscribe", "symbols":["SBER","BTC"] }
server → { "symbol":"SBER","price":305.5,"ts":… }   ← и так на каждый тик
```

## Fan-out: один источник → тысячи получателей

Допустим, открыто 1000 WebSocket-соединений. Тики приходят из
[[Redis — Pub-Sub и кэш|Redis-канала]] `quotes:ticks`. Прямое решение —
**на каждого клиента своя подписка на Redis**. 1000 клиентов = 1000 подписок к
Redis. Это лишняя нагрузка и на Redis, и на сеть.

В проекте используется **fan-out в памяти**: Gateway держит одну подписку на Redis
на весь процесс, а каждый прилетевший тик «разветвляет» всем
WebSocket-клиентам внутри себя. В Kotlin для этого используется
`SharedFlow` — «горячий» поток, на который могут подписаться много слушателей:

```kotlin
// одна подписка на Redis на весь Gateway
private val tickFlow = MutableSharedFlow<Tick>(
    extraBufferCapacity = 8192,
    onBufferOverflow = BufferOverflow.DROP_OLDEST,   // ← см. ниже
)

pubSub.subscribe("quotes:ticks")        // ← ОДИН раз
override fun message(channel, message) {
    tickFlow.tryEmit(tick)              // раздать всем подписчикам Flow
}
```

Каждое WebSocket-соединение читает этот `SharedFlow` и
пере-отправляет клиенту те символы, на которые он подписался.

```
Redis quotes:ticks ──одна подписка──▶ SharedFlow ──┬──▶ клиент 1 (SBER)
                                                    ├──▶ клиент 2 (BTC)
                                                    └──▶ клиент 3 (SBER, GAZP)
```

## Медленный клиент не тормозит остальных

Что если один клиент на медленном мобильном интернете не успевает забирать тики?
Его буфер переполнится. Здесь используется тот же подход, что в драйвере и
[[Go — горутины, каналы и backpressure|quotes-go]]:

При `BufferOverflow.DROP_OLDEST` из заполненного буфера удаляется самый старый
тик. Медленный клиент может пропустить устаревшую цену, но не блокирует
остальные соединения.

Gateway также хранит последний тик каждого символа в `ConcurrentHashMap`. Это
позволяет сразу ответить на `/quotes` и отдать стартовую цену новому
подписчику.
