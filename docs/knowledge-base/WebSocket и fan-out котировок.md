# WebSocket и fan-out котировок

Котировки нужно доставлять клиенту в реальном времени. Обычный HTTP для этого
подходит плохо: клиенту пришлось бы постоянно спрашивать сервер, не появилась
ли новая цена.

## Почему WebSocket

WebSocket открывается один раз и остаётся живым. После этого сервер может сам
отправлять новые цены клиенту. Для мобильного приложения это лучше, чем опрос
по таймеру: меньше лишних запросов и меньше задержка.

В моём случае клиент подписывается на нужные инструменты, а Gateway присылает
только обновления по ним.

```text
client → subscribe: SBER, BTC
server → tick: SBER 305.5
```

## Один поток для многих клиентов

Если у Gateway открыта тысяча WebSocket-подключений, не хочется делать тысячу
подписок на Redis. Поэтому Gateway подписывается на Redis один раз, а потом сам
раздаёт полученные тики всем клиентам.

Это и есть fan-out: один входящий поток превращается во много исходящих.

```text
Redis → Gateway → клиент 1
               → клиент 2
               → клиент 3
```

Так Redis не нагружается количеством мобильных клиентов. Он видит только одну
подписку от экземпляра Gateway.

## Медленные клиенты

Важный момент: медленный клиент не должен тормозить всех остальных. Если у
клиента плохой интернет и он не успевает забирать сообщения, лучше пропустить
старые тики, чем держать очередь бесконечно.

Для котировок это нормальный компромисс. Старая цена быстро теряет смысл,
актуальная важнее.

## Техническая деталь

В Gateway для этого используется `SharedFlow`:

```kotlin
private val tickFlow = MutableSharedFlow<Tick>(
    extraBufferCapacity = 8192,
    onBufferOverflow = BufferOverflow.DROP_OLDEST,
)
```

Каждое WebSocket-соединение читает этот поток и отправляет клиенту только те
символы, на которые он подписался.
