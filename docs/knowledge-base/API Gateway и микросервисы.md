# API Gateway и микросервисы

Серверная часть разделена на несколько сервисов. Клиенты работают с ними через
**API Gateway**. Он служит единой точкой входа и скрывает внутреннее устройство
системы.

## Микросервисы: разные задачи — разные сервисы

Вместо одного приложения — несколько маленьких, каждый со своей зоной
ответственности:

| Сервис | Отвечает за | Стек |
|---|---|---|
| `quotes-go` | сбор котировок из драйвера | Go |
| Data-сервис | счета, ордера, сделки, матчинг | Kotlin/Ktor + PostgreSQL |
| Gateway | вход клиентов, авторизация, раздача котировок | Kotlin/Ktor |

Плюсы такого деления:

- **Разные инструменты под разные задачи.** Сбор тиков — это про конкурентный
  I/O, тут хорош [[Go — горутины, каналы и backpressure|Go]]. Бизнес-логика и
  работа с БД — [[Ktor и корутины Kotlin|Kotlin/Ktor]].
- **Изоляция отказов.** Если упал quotes-go, Data-сервис продолжает работать.
  Развязку
  усиливает [[Redis — Pub-Sub и кэш|шина Redis]]: сервисы общаются через неё, а
  не напрямую.
- **Независимое масштабирование.** Можно поднять 5 копий Gateway и 1 копию
  Data-сервиса — кому сколько нужно.

Плата за это — сложность: сеть между сервисами, отдельные деплои, сложнее
отследить запрос (поэтому есть [[Распределённый трейсинг и OpenTelemetry]]).

## Gateway: единая точка входа

Клиент не ходит во внутренние сервисы напрямую. Все запросы проходят через
Gateway на порту 8080. Он решает несколько общих задач:

- **Аутентификация в одном месте.** Gateway проверяет JWT и достаёт `userId` из
  токена. Внутренние сервисы можно не нагружать проверкой токенов.
- **Сквозные политики.** [[Rate limiting]], CORS, логирование, единый формат
  ошибок — настроены один раз на входе.
- **Скрывает внутреннюю структуру.** Клиент видит один API и не знает,
  что внутри работают три сервиса, Redis и ClickHouse. Реализацию можно
  перекраивать, не трогая клиент.
- **Агрегация.** Gateway собирает ответ из разных источников: котировки из
  [[Redis — Pub-Sub и кэш|Redis]], история свечей из
  [[ClickHouse и колоночные базы данных|ClickHouse]], торговля — проксируется в
  Data-сервис.

## Публичный API vs внутренний API

Тонкий, но важный момент безопасности. Есть **два разных API**:

- **Публичный API** работает через Gateway на порту `8080`. Везде нужен JWT.
  `userId`
  берётся **из токена**.
- **Внутренний API** работает через Data-сервис на порту `8082` и доступен
  только Gateway. Эндпоинты вида
  `/internal/orders` принимают `userId` **параметром** и не проверяют токены.

Почему так можно: внутренний сервис **не торчит наружу** (только в приватной
docker-сети). Gateway — доверенный посредник: он уже проверил токен и подставляет
`userId` сам. Клиент физически не может постучать в `:8082` и притвориться чужим
пользователем.

Эта схема безопасна, только пока порт `8082` недоступен извне. Поэтому
Data-сервис подключён к внутренней Docker-сети.

Gateway частично выполняет роль Backend for Frontend. Например, он возвращает
`changePct` и подготавливает данные для экранов клиента.
