# API Gateway и микросервисы

Вход в систему вынесен в отдельный Gateway, чтобы клиент не зависел от
внутреннего устройства серверной части. Для приложения это выглядит как один
сервер на `:8080`, а внутри уже есть Data-сервис, генератор котировок, Redis и
ClickHouse.

Такая схема хорошо подходит проекту, потому что у разных частей системы разные
задачи. Котировки живут своим потоком, торговые операции требуют аккуратной
работы с БД, а мобильному клиенту нужен стабильный и понятный API.

## Общая схема

Gateway — это входная дверь. Через неё проходят запросы пользователя,
авторизация, лимиты и WebSocket с котировками. Внутренние сервисы наружу не
выставляются.

Получается такая картина:

| Часть | Что делает |
|---|---|
| `quotes-go` | генерирует и публикует котировки |
| Data-сервис | работает со счетами, заявками и сделками |
| Gateway | принимает запросы клиента и проксирует их дальше |

Главный плюс — клиент не зависит от внутренней архитектуры. Если поменять способ
хранения истории котировок или разделить Data-сервис ещё на части,
Android-приложение не должно из-за этого переписываться.

Минус тоже понятный: сервисов становится больше, и обычный запрос сложнее
отследить. Поэтому рядом появляется [[Распределённый трейсинг и OpenTelemetry]].

## Что делает Gateway

В проекте Gateway отвечает за общие вещи:

- проверяет токен и достаёт пользователя;
- применяет [[Rate limiting]];
- отдаёт котировки через [[WebSocket и fan-out котировок]];
- ходит во внутренний Data-сервис;
- собирает данные для клиента в удобном виде.

Важно, что `userId` берётся из токена, а не из тела запроса. Клиент не может
передать чужой идентификатор и получить чужие данные.

## Техническая деталь

Публичный API работает через Gateway:

```text
Android → Gateway :8080 → Data-сервис :8082
```

Data-сервис доступен только во внутренней Docker-сети. Поэтому эндпоинты вида
`/internal/orders` можно считать внутренними: туда ходит Gateway после проверки
токена.
