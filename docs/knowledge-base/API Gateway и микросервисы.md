# API Gateway и микросервисы

Систему можно было написать одним большим приложением (монолитом). Вместо этого
она разрезана на несколько **микросервисов**, а перед ними стоит **API
Gateway** — единственная дверь для клиентов. Разберём, зачем так и кто за что
отвечает.

## Микросервисы: разные задачи — разные сервисы

Вместо одного приложения — несколько маленьких, каждый со своей зоной
ответственности:

| Сервис | Отвечает за | Стек |
|---|---|---|
| `#6` quotes-go | сбор котировок из драйвера | Go |
| `#4` Data-сервис | счета, ордера, сделки, матчинг | Kotlin/Ktor + PostgreSQL |
| `#3` Gateway | вход клиентов, auth, раздача котировок | Kotlin/Ktor |

Плюсы такого деления:

- **Разные инструменты под разные задачи.** Сбор тиков — это про конкурентный
  I/O, тут хорош [[Go — горутины, каналы и backpressure|Go]]. Бизнес-логика и
  работа с БД — [[Ktor и корутины Kotlin|Kotlin/Ktor]].
- **Изоляция отказов.** Упал quotes-go — торговля (Data-сервис) живёт. Развязку
  усиливает [[Redis — Pub-Sub и кэш|шина Redis]]: сервисы общаются через неё, а
  не напрямую.
- **Независимое масштабирование.** Можно поднять 5 копий Gateway и 1 копию
  Data-сервиса — кому сколько нужно.

Плата за это — сложность: сеть между сервисами, отдельные деплои, сложнее
отследить запрос (поэтому есть [[Распределённый трейсинг и OpenTelemetry]]).

## Gateway: единая точка входа

Клиент (Android, нагрузочный имитатор) **не ходит в сервисы напрямую** — только
через Gateway на порту 8080. Зачем нужен этот «привратник»:

- **Аутентификация в одном месте.** Gateway проверяет [[JWT, токены и bcrypt|JWT]]
  и достаёт `userId` из токена. Внутренние сервисы можно не нагружать auth.
- **Сквозные политики.** [[Rate limiting]], CORS, логирование, единый формат
  ошибок — настроены один раз на входе.
- **Скрывает внутреннюю кухню.** Клиент видит один аккуратный API и не знает,
  что под капотом три сервиса, Redis и ClickHouse. Внутренности можно
  перекраивать, не трогая клиент.
- **Агрегация.** Gateway собирает ответ из разных источников: котировки из
  [[Redis — Pub-Sub и кэш|Redis]], история свечей из
  [[ClickHouse и колоночные базы данных|ClickHouse]], торговля — проксируется в
  Data-сервис.

## Публичный API vs внутренний API

Тонкий, но важный момент безопасности. Есть **два разных API**:

- **Публичный** (Gateway, `:8080`) — для клиентов. Везде нужен JWT. `userId`
  берётся **из токена**.
- **Внутренний** (Data-сервис, `:8082`) — только для Gateway. Эндпоинты вида
  `/internal/orders` принимают `userId` **параметром** и не проверяют токены.

Почему так можно: внутренний сервис **не торчит наружу** (только в приватной
docker-сети). Gateway — доверенный посредник: он уже проверил токен и подставляет
`userId` сам. Клиент физически не может постучать в `:8082` и притвориться чужим
пользователем.

> [!warning] Это работает, только если внутренний порт закрыт
> Вся модель держится на том, что `:8082` недоступен извне. Открой его в
> интернет — и любой сможет дёргать `/internal/orders?userId=любой`. В
> docker-compose Data-сервис живёт во внутренней сети именно поэтому.

> [!note] Gateway здесь ещё и BFF
> Поскольку Gateway заточен под конкретных клиентов (отдаёт котировки с
> `changePct`, агрегирует свечи под графики), он по сути ещё и **Backend for
> Frontend** — слой, который подгоняет данные под нужды UI.
