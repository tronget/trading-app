# Rate limiting

Rate limiting нужен, чтобы один клиент не мог забить Gateway запросами. Это
простая защита от перегрузки: если запросов слишком много, сервер отвечает
`429 Too Many Requests`.

## Зачем я это добавил

Даже в учебном проекте легко получить ситуацию, когда клиент случайно уходит в
цикл запросов. Например, ошибка в retry-логике или слишком частое обновление
экрана. Без ограничения такой клиент будет мешать всем остальным.

Лимит не делает систему полностью защищённой, но добавляет понятный
предохранитель на входе. В проекте это работает вместе с
[[HikariCP и пул соединений|пулом соединений к БД]] и ограничениями в
[[WebSocket и fan-out котировок|раздаче котировок]].

## Как работает ограничение

Для каждого клиента есть свой счётчик запросов. Если он делает запросы в
допустимом темпе, всё проходит. Если темп слишком высокий, Gateway начинает
отбивать лишние запросы.

Важно считать лимит отдельно для разных клиентов. Иначе один шумный клиент мог
бы исчерпать общий лимит и заблокировать всех.

## Техническая деталь

В Ktor это настраивается через `RateLimit`:

```kotlin
install(RateLimit) {
    register(RateLimitName("api")) {
        rateLimiter(limit = config.rateLimitPerSecond, refillPeriod = 1.seconds)
        requestKey { call -> call.request.origin.remoteHost }
    }
}
```

Если Gateway будет стоять за прокси, реальный IP надо брать аккуратно: только
из заголовков доверенного балансировщика, а не из любого значения от клиента.
