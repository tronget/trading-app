# Rate limiting

**Rate limiting** ограничивает частоту запросов. Если клиент превышает лимит,
Gateway возвращает `429 Too Many Requests`.

## Зачем это нужно

- **Защита от перегрузки.** Один кривой клиент в цикле, retry-шторм или просто
  всплеск трафика не должны положить сервис для всех остальных.
- **Защита от абьюза.** Перебор паролей на `/auth/login`, парсинг данных пачками —
  всё это упирается в лимит.
- **Справедливость.** Ресурсы делятся между клиентами, один не съедает все.

Это часть «защиты от перегрузки» наряду с
[[WebSocket и fan-out котировок|DROP_OLDEST в fan-out]] и
[[HikariCP и пул соединений|лимитом пула соединений к БД]] — у каждого
слоя свой предохранитель.

## Token bucket — как считают лимит

Ktor использует алгоритм **token bucket**:

- есть ведро на N жетонов;
- каждый запрос съедает жетон;
- жетоны **добавляются** с постоянной скоростью;
- ведро пустое → запрос отбивается (`429`).

Такой алгоритм разрешает короткие всплески, но ограничивает среднюю скорость.

```kotlin
install(RateLimit) {
    register(RateLimitName("api")) {
        rateLimiter(limit = config.rateLimitPerSecond, refillPeriod = 1.seconds)
        requestKey { call -> call.request.origin.remoteHost }   // ключ = IP
    }
}
```

## Лимит на каждый IP отдельно

Ключевая деталь — **`requestKey = remoteHost`**: ведро **своё на каждый IP**, а
не одно общее. Иначе один шумный клиент исчерпал бы общий лимит и заблокировал
всех. Так у каждого клиента свой счётчик.

Когда лимит превышен, Ktor возвращает `429`, а
[[Ktor и корутины Kotlin|StatusPages]] оборачивает это в аккуратный JSON:

```kotlin
status(HttpStatusCode.TooManyRequests) { call, status ->
    call.respond(status, ErrorResponse("rate limit exceeded"))
}
```

Текущий лимит считается отдельно на каждом экземпляре Gateway. Если экземпляров
станет несколько, общий счётчик можно вынести в Redis.

За прокси `remoteHost` может содержать адрес самого прокси. Тогда реальный IP
берётся из `X-Forwarded-For`, но только от доверенного балансировщика.

Нагрузочный имитатор отправляет запросы в пределах настроенного лимита. Его
задача — измерить задержки и пропускную способность, а не проверить защиту от
частых запросов.
