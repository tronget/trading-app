# Распределённый трейсинг и OpenTelemetry

Когда запрос проходит через несколько сервисов, обычного лога уже мало. В логе
видно отдельные события, но трудно понять, где именно запрос потерял время.
Для этого используется трейсинг.

## Зачем он нужен

Например, пользователь отправил заявку. Запрос пришёл в Gateway, потом ушёл в
Data-сервис, потом в PostgreSQL. Если ответ был медленным, нужно понять, где
задержка: на входе, в сети, в бизнес-логике или в SQL.

Трейсинг показывает один запрос как цепочку операций. Сразу видно, какая часть
сколько заняла.

```text
POST /orders
  └─ data-service
       └─ db.tx place_order
            └─ SELECT ... FOR UPDATE
```

## Основные понятия

Спан — это один кусок работы. Трейс — набор таких кусков для одного запроса.
Если всё склеено правильно, можно открыть Jaeger и увидеть весь путь запроса.

Это особенно полезно в микросервисах. Без трейсинга легко начать гадать:
«кажется, тормозит Gateway» или «наверное, база». С трейсингом видно точнее.

## Что оказалось важным

Нужно передавать контекст между сервисами. Gateway должен отправить специальный
заголовок дальше, чтобы Data-сервис понял: это продолжение того же запроса, а
не новый независимый трейс.

В Kotlin с корутинами есть ещё одна тонкость. Корутина может продолжиться на
другом потоке, поэтому контекст текущего спана нужно переносить аккуратно. Иначе
дочерние операции могут оторваться от родительского запроса.

## Техническая деталь

Контекст передаётся через `traceparent`:

```text
traceparent: 00-<trace_id>-<span_id>-01
```

Сервисы отправляют данные в OpenTelemetry Collector, а смотреть результат можно
в Jaeger на `localhost:16686`.

Метрики и логи описаны отдельно в [[06 — Наблюдаемость]].
