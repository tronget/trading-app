# Распределённый трейсинг и OpenTelemetry

Один запрос может пройти через Gateway, Data-сервис и PostgreSQL. Чтобы увидеть,
где он задержался, используется распределённый трейсинг на базе
**OpenTelemetry и Jaeger**.

## Спаны и трейсы

Два понятия:

- **Спан** — один отрезок работы: «HTTP-запрос на Gateway», «SQL-транзакция
  `place_order`», «публикация в Redis». У спана есть имя, время начала/конца и
  статус (ок/ошибка).
- **Трейс** — дерево спанов одного запроса. Корневой спан —
  родитель, вложенные операции — дети.

В интерфейсе видно, что запрос занял 120 мс, из них 90 мс ушло на
SQL-транзакцию, а внутри задержался конкретный `SELECT`.

```
trace ──┬─ gateway: POST /orders                 120ms
        └─ data-service: db.tx place_order         95ms
             └─ SELECT … FOR UPDATE                70ms   ← вот он, тормоз
```

## Как трейс «склеивается» между сервисами

Каждый трейс имеет `trace_id`. Чтобы спан в
Data-сервисе попал в **то же** дерево, что и спан Gateway, нужно протащить
`trace_id` по сети. Это делает стандарт **W3C Trace Context** — HTTP-заголовок
`traceparent`:

```
traceparent: 00-<trace_id>-<span_id>-01
```

Gateway создаёт серверный спан на запрос и **инжектирует `traceparent`** в вызов
Data-сервиса. Data-сервис **извлекает** его и вешает свои спаны как **детей**
этого `trace_id`. Так три сервиса собирают один общий трейс. quotes-go тоже
ставит спаны — на публикацию в [[Redis — Pub-Sub и кэш|Redis]] и батч-вставку в
[[ClickHouse и колоночные базы данных|ClickHouse]].

## Контекст в корутинах

В обычном коде текущий спан часто хранится в
**thread-local** — переменной, привязанной к потоку. Но в
[[Ktor и корутины Kotlin|корутинах]] одна задача в течение жизни **прыгает между
потоками** (приостановилась на одном — возобновилась на другом). Thread-local
при этом теряется, и дочерний спан «отрывается» от родителя.

Решение — переносить контекст вместе с корутиной через
`opentelemetry-extension-kotlin` (`asContextElement`). В `db.tx` это выглядит так:

```kotlin
val span = tracer.spanBuilder("db.tx $name").startSpan()
span.makeCurrent().use {
    dataSource.connection.use { c -> … }   // спан остаётся «текущим» внутри
}
span.end()
```

В корутинном коде контекст спана нужно переносить вместе с корутиной. Иначе
дочерние операции попадут в отдельные трейсы.

## OpenTelemetry, Collector, Jaeger — кто есть кто

- **OpenTelemetry** — независимый от конкретного хранилища стандарт и набор
  библиотек для спанов и метрик.
- **OTel Collector** — труба-посредник: сервисы шлют телеметрию в него
  (`OTEL_EXPORTER_OTLP_ENDPOINT`), он пересылает дальше. Удобно — можно сменить
  бэкенд, не трогая сервисы.
- **Jaeger** — хранилище и UI для трейсов. Открываешь http://localhost:16686,
  ищешь трейс, смотришь водопад.

Метрики и логи — соседние темы, см. [[06 — Наблюдаемость]].
