# HikariCP и пул соединений

HikariCP относится не к бизнес-логике, а к инфраструктуре работы с PostgreSQL.
Основная идея — не открывать новое соединение с БД на каждый запрос.

## Зачем это в проекте

Открытие соединения с PostgreSQL занимает время: сетевое подключение,
аутентификация и настройка сессии. Если делать это постоянно, задержки будут
выше, а база начнёт тратить ресурсы не на запросы, а на подключения.

Пул держит несколько готовых соединений. Код берёт соединение, выполняет работу
и возвращает его обратно. Для сервиса это выглядит как обычное получение
соединения, но под капотом оно переиспользуется.

## Почему размер пула важен

Размер пула — это не случайное число. Если поставить слишком мало соединений,
запросы будут ждать очередь. Если слишком много — можно перегрузить PostgreSQL.

Поэтому пул надо смотреть вместе с нагрузочным тестом. Если p99 растёт, причина
может быть не в HTTP и не в Kotlin, а в том, что запросы ждут свободное
соединение или слишком долго держат его занятым.

## Техническая деталь

Пример конфигурации:

```kotlin
val dataSource = HikariDataSource(HikariConfig().apply {
    jdbcUrl = url
    username = user
    password = password
    maximumPoolSize = 20
    minimumIdle = 2
})
```

`maximumPoolSize` ограничивает число одновременных соединений к базе. Это ещё
один предохранитель от перегрузки, рядом с [[Rate limiting]] и ограничениями в
самом сервисе.
