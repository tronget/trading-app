# HikariCP и пул соединений

В Data-сервисе запросы к PostgreSQL идут через пул соединений HikariCP.
Это не отдельная бизнес-логика, а инфраструктурная часть: сервису нужно быстро
получать соединение с БД и не открывать новое TCP-соединение на каждый запрос.

## Зачем нужен пул

Открыть соединение к PostgreSQL дорого: есть сетевое соединение,
аутентификация и настройка сессии. Если делать это на каждый HTTP-запрос,
задержка вырастет, а база будет тратить ресурсы на постоянные подключения и
отключения.

Пул держит несколько уже открытых соединений и переиспользует их:

```kotlin
val dataSource = HikariDataSource(HikariConfig().apply {
    jdbcUrl = url
    username = user
    password = password
    maximumPoolSize = 20
    minimumIdle = 2
})
```

Когда код берёт `dataSource.connection`, HikariCP выдаёт свободное соединение
из пула. Когда вызывается `close()`, соединение не закрывается физически, а
возвращается обратно в пул.

## Почему размер пула важен

`maximumPoolSize = 20` задаёт верхнюю границу одновременных соединений к
PostgreSQL. Это полезно сразу по двум причинам:

- сервис не создаёт бесконечное число подключений при всплеске запросов;
- база получает предсказуемую нагрузку, а лишние запросы ждут свободное
  соединение.

Если пул слишком маленький, запросы начнут стоять в очереди. Если слишком
большой — можно перегрузить PostgreSQL большим числом параллельных операций.
Поэтому размер пула подбирается вместе с нагрузочным тестированием, а не
ставится «на глаз».

## Что важно в проекте

Для торговых операций важна стабильная задержка. HikariCP помогает убрать
лишние расходы на открытие соединений, но сам по себе не ускоряет медленный
SQL. Если запрос долго держит соединение, оно всё это время занято, и очередь
к пулу растёт.

Поэтому пул соединений надо смотреть вместе с p95/p99 задержками и метриками
PostgreSQL. Если во время теста растёт ожидание свободного соединения, проблема
может быть не в HTTP-слое, а в БД или размере пула.
