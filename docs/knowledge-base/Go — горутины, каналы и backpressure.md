# Go: горутины, каналы и backpressure

Сборщик котировок `quotes-go` и нагрузочный имитатор написаны на **Go**. В обоих
случаях нужно выполнять много операций ввода-вывода одновременно.

## Горутины — дешёвые «потоки»

Горутина — это функция, запущенная конкурентно, словом `go`:

```go
go readDevice("/dev/stock0")   // поехала в фоне, не блокируя нас
```

Горутины требуют меньше памяти, чем обычные потоки ОС. Горутина стартует с
небольшим стеком и растёт по
необходимости, а планировщик Go сам раскладывает тысячи горутин на горстку
системных потоков. Поэтому имитатор спокойно поднимает **10 000 «клиентов» = 10
000 горутин** (см. [[Нагрузочное тестирование и перцентили]]), а quotes-go
держит по горутине на каждое устройство-источник.

Когда горутина упирается в блокирующую операцию (чтение из `/dev/stock0`,
сетевой запрос), Go-рантайм просто снимает её с потока и ставит другую. Поэтому
«заблокироваться на чтении» здесь дёшево — в отличие от классической модели,
где блокированный поток простаивает зря.

## Каналы — обмен данными без общей памяти

Горутины общаются через **каналы** — типизированные очереди:

```go
ticks := make(chan Tick, 1024)   // буфер на 1024 тика
ticks <- tick                    // отправить
t := <-ticks                     // получить
```

Вместо общей изменяемой структуры данные можно передавать через канал. Так в
quotes-go устроена раздача данных: горутина-читатель устройства отправляет тик
в Redis, ClickHouse и SSE. У каждого получателя свой канал.

## Backpressure: что делать, когда не успеваешь

Тики идут постоянно. Если запись в ClickHouse замедлилась, есть три варианта:

1. **Блокироваться** — ждать, пока потребитель разгребёт. Но тогда тормозит
   весь конвейер, включая быстрых потребителей. Плохо.
2. **Копить в бесконечном буфере** — рано или поздно съест всю память. Тоже плохо.
3. **Выкинуть старое** — отправить в канал «неблокирующе», и если он полон —
   дропнуть тик. Для котировок это нормально: устаревшая цена никому не нужна,
   важна свежая.

Проект выбирает вариант 3 — **неблокирующая отправка**:

```go
select {
case ch <- tick:           // влезло — отлично
default:                    // канал полон — дропаем, не ждём
    metrics.dropped++
}
```

Это и есть управление **backpressure** (обратным давлением): медленный
потребитель не имеет права тормозить остальных. Дропнутые тики считаются метрикой
`quotes_ticks_dropped_total` (см. [[06 — Наблюдаемость]]). Тот же принцип
используется в кольцевом буфере драйвера и в
[[WebSocket и fan-out котировок|SharedFlow Gateway]].

Для отладки quotes-go также отдаёт тики через **Server-Sent Events**. Это
обычный HTTP-ответ, который остаётся открытым и передаёт новые строки `data: …`.
