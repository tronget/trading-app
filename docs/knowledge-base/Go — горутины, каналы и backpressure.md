# Go: горутины, каналы и backpressure

Go в проекте нужен там, где много параллельного ввода-вывода. Это генератор
котировок `quotes-go` и нагрузочный имитатор. Выбор Go здесь связан не с самим
языком, а с тем, что он удобно решает задачу с большим числом одновременных
действий.

## Почему горутины подошли

Горутина — это лёгкая задача. Её можно запустить почти как обычную функцию, но
она будет выполняться параллельно:

```go
go readDevice("/dev/stock0")
```

В нагрузочном тесте это особенно удобно. Один виртуальный клиент — одна
горутина. Поэтому 10 000 клиентов не превращаются в 10 000 тяжёлых потоков ОС.

Для `quotes-go` смысл похожий: один кусок кода читает котировки, другой пишет в
Redis, третий отправляет данные дальше. Это проще держать в голове, чем большой
общий цикл со множеством состояний.

## Что делать, если кто-то не успевает

Самая важная часть здесь — не горутины сами по себе, а ситуация, когда данные
приходят быстрее, чем их успевают обработать.

Например, ClickHouse на секунду стал медленнее. Нельзя из-за этого остановить
всю раздачу котировок. Для котировок лучше потерять старый тик, чем задерживать
свежий.

Поэтому в проекте используется подход: если буфер заполнен, старые данные можно
выкинуть. Для котировок это допустимый компромисс: пользователю важнее
актуальная цена сейчас, а не каждая промежуточная точка.

## Техническая деталь

В Go это выглядит через неблокирующую отправку в канал:

```go
select {
case ch <- tick:
    // тик попал в канал
default:
    metrics.dropped++
}
```

Если канал полон, код не ждёт. Он считает дроп и идёт дальше. Такой же принцип
используется в [[WebSocket и fan-out котировок]], где медленный клиент не должен
тормозить остальных.
