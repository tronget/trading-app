# Голый SQL, JDBC и пул соединений

В Data-сервисе (`#4`) **нет ORM** — никакого Hibernate/Exposed. Все запросы
написаны руками на SQL через **JDBC**. Это требование задания, но заодно и
хороший повод понять, что ORM от тебя обычно прячет.

## Что такое ORM и почему здесь без него

ORM (Object-Relational Mapping) — библиотека, которая превращает строки таблиц в
объекты и генерирует SQL за тебя: пишешь `user.save()`, а она сама придумывает
`INSERT`. Удобно, но:

- прячет реальный SQL — легко породить медленные запросы, не заметив;
- свои абстракции над транзакциями и блокировками (а нам нужен точный контроль —
  см. [[PostgreSQL — транзакции, блокировки и матчинг]]);
- «магия», которую надо отдельно изучать.

Голый SQL — многословнее, зато ты **видишь ровно то, что уходит в БД**. Для
финансовой логики, где важна каждая блокировка, это плюс.

## JDBC: как Kotlin/Java говорит с БД

JDBC — стандартный Java-API к реляционным БД. Базовый цикл: взять соединение →
подготовить запрос → подставить параметры → выполнить → разобрать результат.

```kotlin
fun findByLogin(c: Connection, login: String): UserWithHashDto? =
    c.prepareStatement("SELECT id, login, password_hash FROM users WHERE login = ?").use { st ->
        st.setString(1, login)          // параметр вместо ?
        st.executeQuery().singleOrNull { rs ->
            UserWithHashDto(rs.getLong("id"), rs.getString("login"), rs.getString("password_hash"))
        }
    }
```

Два важных момента:

- **`prepareStatement` + `?`** — это *параметризованный* запрос. Значение
  передаётся отдельно от текста SQL, поэтому **SQL-инъекция невозможна** в
  принципе: что бы пользователь ни ввёл в `login`, это останется данными, а не
  кодом. Склейка строк (`"… WHERE login = '$login'"`) — так делать нельзя
  никогда.
- **`.use { }`** — Kotlin-аналог try-with-resources: соединение/стейтмент
  гарантированно закроются, даже если выскочит исключение. Иначе утечёшь
  ресурсы и положишь пул.

## Пул соединений (HikariCP)

Открыть TCP-соединение к PostgreSQL — дорого (рукопожатие, аутентификация). При
тысячах запросов в секунду открывать его каждый раз — самоубийство. Поэтому
используется **пул соединений**: заранее открытая пачка соединений, которые
переиспользуются.

```kotlin
val dataSource = HikariDataSource(HikariConfig().apply {
    jdbcUrl = url; username = user; password = password
    maximumPoolSize = 20    // не больше 20 соединений к БД
    minimumIdle = 2
})
```

**HikariCP** — самый быстрый пул в JVM-мире. `dataSource.connection` выдаёт
соединение из пула, `.close()` (через `.use`) **возвращает его обратно**, а не
рвёт. `maximumPoolSize = 20` ещё и защищает саму БД: сколько бы ни пришло
запросов, к Postgres одновременно постучатся максимум 20 — остальные подождут в
очереди (см. также [[Нагрузочное тестирование и перцентили]]).

## Главный приём: репозитории на `Connection`

Вот ради чего всё затевалось. Каждая функция-репозиторий **принимает
`Connection` снаружи**, а не берёт своё:

```kotlin
object AccountRepo {
    fun lockCash(c: Connection, userId: Long): BigDecimal? = …
    fun addCash(c: Connection, userId: Long, delta: BigDecimal) = …
}
```

Почему это важно: несколько вызовов репозиториев на **одном и том же
`Connection`** автоматически попадают в **одну транзакцию**. Сервис компонует их
как кубики:

```kotlin
db.tx("place_order") { c ->      // одно соединение = одна транзакция
    val cash = AccountRepo.lockCash(c, userId)   // блокировка
    PositionRepo.upsert(c, …)                    // те же скобки транзакции
    AccountRepo.addCash(c, userId, cost.negate())
    TradeRepo.insert(c, …)
}                                 // commit/rollback здесь
```

А `db.tx` — это тонкая обёртка: берёт соединение из пула, ставит
`autoCommit = false`, выполняет блок, делает `commit()` (или `rollback()` при
исключении) и заодно вешает спан [[Распределённый трейсинг и OpenTelemetry|трейсинга]].
Просто, прозрачно, без магии ORM.

> [!tip] Вывод
> Передавать `Connection` в репозитории — это и есть способ собирать
> транзакции из мелких операций, когда нет ORM. Репозиторий ничего не знает про
> commit — он просто «пишет в то соединение, что дали».
