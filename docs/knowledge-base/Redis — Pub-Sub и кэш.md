# Redis: Pub/Sub и кэш

В проекте **Redis** используется для двух задач: передаёт новые тики через
Pub/Sub и хранит последнюю цену каждого инструмента.

## Роль 1: Pub/Sub — шина тиков

Pub/Sub работает по модели издателя и подписчиков:

- `quotes-go` **публикует** каждый тик в канал `quotes:ticks`;
- кто подписан на канал — **получает** сообщение; кто не подписан — нет.

```
quotes-go ──PUBLISH quotes:ticks──▶ Redis ──▶ Gateway   (раздаёт в WebSocket)
                                          └──▶ Data-сервис (матчинг лимиток)
```

Издатель ничего не знает о подписчиках. Gateway и Data-сервис подписаны
независимо. Новый потребитель можно добавить без изменений в quotes-go.

Redis Pub/Sub не хранит сообщения. Если подписчик был отключён, он пропустит
тик. Для котировок это допустимо: скоро придёт новая цена, а история всё равно
записывается в [[ClickHouse и колоночные базы данных|ClickHouse]]. Торговые
операции через Pub/Sub не передаются. Они идут по HTTP в Data-сервис.

### Почему здесь Redis, а не Kafka

Kafka/RabbitMQ дают persistence, повторную доставку, группы потребителей — но
это лишняя сложность для учебного проекта, где поток тиков эфемерен. Redis уже
есть в стеке (он же кэш), он простой и очень быстрый. Берём то, что подходит
по задаче, а не «самое серьёзное».

## Роль 2: кэш последней цены

Параллельно с публикацией quotes-go кладёт последний тик каждого символа в
обычный ключ:

```
SET quote:last:SBER   {"symbol":"SBER","price":305.5,"ts":...}
```

Pub/Sub доставляет только новые сообщения активным подписчикам. Для торговой
операции текущая цена нужна сразу, поэтому Data-сервис читает
`quote:last:<symbol>` — мгновенно, из памяти. Так заявка может использовать
последнюю известную цену:

```kotlin
val price = prices.lastPrice(order.symbol)
    ?: return rejected   // цены ещё нет — отклоняем
```

Gateway делает то же для списка котировок, плюс держит ещё и **локальный**
кэш в памяти (`ConcurrentHashMap`), а в Redis ходит только при промахе — лишний
сетевой round-trip ни к чему.

В итоге Pub/Sub отвечает за поток обновлений, а обычный ключ Redis — за
последнее известное состояние.
