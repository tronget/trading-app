# Redis: Pub/Sub и кэш

**Redis** — это быстрое хранилище «ключ-значение» в оперативной памяти. В
проекте он работает сразу в двух ролях: как **шина сообщений** (доставляет тики)
и как **кэш** (хранит последнюю цену). Разберём обе.

## Роль 1: Pub/Sub — шина тиков

Pub/Sub (publish/subscribe, издатель/подписчик) — это модель «вещания по
радиоканалу»:

- `quotes-go` (`#6`) **публикует** каждый тик в канал `quotes:ticks`;
- кто подписан на канал — **получает** сообщение; кто не подписан — нет.

```
quotes-go ──PUBLISH quotes:ticks──▶ Redis ──▶ Gateway   (раздаёт в WebSocket)
                                          └──▶ Data-сервис (матчинг лимиток)
```

Прелесть в **развязке** (decoupling): издатель ничего не знает о подписчиках.
Gateway и Data-сервис подписаны независимо; упал один — второй работает; добавим
третьего потребителя — издателя менять не нужно. Это снимает жёсткие связи между
сервисами (см. [[API Gateway и микросервисы]]).

> [!warning] Подводный камень Pub/Sub: «выстрелил и забыл»
> Redis Pub/Sub — это **fire-and-forget**, доставка «максимум один раз». Если
> подписчик в момент публикации отключён — сообщение для него **потеряно**, оно
> нигде не копится. Для котировок это ОК (свежий тик прилетит через полсекунды,
> а историю всё равно хранит [[ClickHouse и колоночные базы данных|ClickHouse]]).
> Но для денег/заказов так нельзя — там нужна гарантированная доставка
> (Kafka/RabbitMQ/очереди с подтверждением). Поэтому **торговые операции идут не
> через Pub/Sub, а через обычный HTTP** в Data-сервис.

### Почему здесь Redis, а не Kafka

Kafka/RabbitMQ дают persistence, повторную доставку, группы потребителей — но
это лишняя сложность для учебного проекта, где поток тиков эфемерен. Redis уже
есть в стеке (он же кэш), он простой и очень быстрый. Берём то, что подходит
по задаче, а не «самое серьёзное».

## Роль 2: кэш последней цены

Параллельно с публикацией quotes-go кладёт последний тик каждого символа в
обычный ключ:

```
SET quote:last:SBER   {"symbol":"SBER","price":305.5,"ts":...}
```

Зачем дублировать? Pub/Sub доставляет только **новые** тики тем, кто **сейчас**
слушает. А что если market-ордер пришёл прямо сейчас и нужна текущая цена,
**не дожидаясь** следующего тика? Тогда Data-сервис просто читает
`quote:last:<symbol>` — мгновенно, из памяти. Так
[[Ордера, позиции и P&L|market-ордер]] исполняется по последней известной цене:

```kotlin
val price = prices.lastPrice(order.symbol)
    ?: return rejected   // цены ещё нет — отклоняем
```

Gateway делает то же для списка котировок, плюс держит ещё и **локальный**
кэш в памяти (`ConcurrentHashMap`), а в Redis ходит только при промахе — лишний
сетевой round-trip ни к чему.

> [!tip] Pattern: «шина + кэш»
> Связка «Pub/Sub для потока событий + key-value для последнего состояния» —
> классика для real-time данных. Подписчик получает обновления, а свежий пришёл
> позже — всегда может спросить «а какое сейчас значение?» у кэша.
