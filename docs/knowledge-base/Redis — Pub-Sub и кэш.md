# Redis: Pub/Sub и кэш

Redis в проекте выполняет две простые роли: передаёт новые котировки между
сервисами и хранит последнюю цену каждого инструмента.

## Зачем здесь Redis

Котировки генерирует `quotes-go`, но нужны они не только ему. Gateway должен
отправлять их клиентам по WebSocket, а Data-сервису нужна актуальная цена для
торговых операций.

Можно было бы связать сервисы напрямую, но тогда они стали бы сильнее зависеть
друг от друга. Redis здесь работает как промежуточная шина: один сервис
публикует тик, остальные получают его, если подписаны.

## Почему это подходит для котировок

Redis Pub/Sub не хранит сообщения навсегда. Если сервис был выключен, он
пропустит часть тиков. Для котировок это приемлемо: скоро придёт новая цена, а
история всё равно пишется в [[ClickHouse и колоночные базы данных|ClickHouse]].

Торговые операции через Pub/Sub не идут. Они проходят обычными HTTP-запросами,
потому что там важна надёжность и понятный ответ.

## Последняя цена

Кроме потока тиков, Redis хранит последнее известное значение:

```text
quote:last:SBER → {"symbol":"SBER","price":305.5,"ts":...}
```

Это удобно, когда новому клиенту нужно сразу показать текущие цены, не дожидаясь
следующего тика. Gateway может быстро взять последнее значение и уже потом
подписать клиента на поток обновлений.

## Техническая деталь

Схема получается такая:

```text
quotes-go → Redis Pub/Sub → Gateway → WebSocket-клиенты
                       └→ Data-сервис
```

Gateway ещё держит локальный кэш в памяти, чтобы не ходить в Redis на каждый
запрос списка котировок.
