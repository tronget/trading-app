# Jetpack Compose, MVVM и StateFlow

Android-приложение написано на **Jetpack Compose**. Состояние экранов хранится
во ViewModel и передаётся в UI через **StateFlow**.

## Jetpack Compose: декларативный UI

В классическом Android UI описывали XML-разметкой, а элементы вручную меняли из
кода (`textView.text = "..."`). При таком подходе экран проще рассинхронизировать
с данными.

Compose работает декларативно. Мы описываем экран для текущего состояния, а
при изменении данных Compose обновляет нужную часть интерфейса.

```kotlin
@Composable
fun MarketScreen(state: MarketState) {
    LazyColumn {
        items(state.quotes) { q ->
            Text("${q.symbol}  ${q.last}")   // меняется q.last → строка сама обновится
        }
    }
}
```

Функции с `@Composable` — это «кирпичики» UI. Когда данные, которые они читают,
меняются, Compose делает **рекомпозицию** — перевызывает только затронутые
кусочки и перерисовывает минимально нужное. Никаких ручных `findViewById` и
обновлений «руками».

## MVVM: кто за что отвечает

**MVVM** разделяет экран на три слоя:

- **Model** — данные и бизнес-логика (репозитории, сетевой клиент
  [[DataStore и авто-refresh токена|ApiClient]], модели).
- **View** — `@Composable`-функции. Только рисуют состояние и шлют события
  («нажали Купить»). Никакой логики.
- **ViewModel** — посредник: держит состояние экрана, обрабатывает события,
  ходит в Model, обновляет состояние. Переживает поворот экрана (не пересоздаётся
  вместе с Activity).

```
Экран Compose ──события──▶ ViewModel ──▶ репозиторий и API
      ▲                         │
      └────── состояние ────────┘
```

View не знает, откуда пришли данные, а Model не зависит от UI. Поэтому логику
можно тестировать отдельно, а экран можно открыть с тестовым состоянием.

## StateFlow: состояние как поток

Как ViewModel отдаёт состояние во View, чтобы та сама перерисовывалась?
Через **StateFlow** — «наблюдаемый контейнер с текущим значением»:

```kotlin
class MarketViewModel(...) : ViewModel() {
    private val _state = MutableStateFlow(MarketState())
    val state: StateFlow<MarketState> = _state      // наружу — только для чтения

    fun onTick(tick: Tick) {
        _state.update { it.copy(quotes = it.quotes.updatePrice(tick)) }  // новое состояние
    }
}
```

А Compose подписывается на него и автоматически рекомпозит UI при изменении:

```kotlin
val state by viewModel.state.collectAsStateWithLifecycle()
MarketScreen(state)
```

Свойства StateFlow, которые тут важны:

- всегда есть **текущее значение** (новый подписчик сразу его получает);
- отдаёт **только при изменении** — если новое значение равно старому, лишней
  рекомпозиции не будет;
- данные движутся в одном направлении: состояние идёт из ViewModel в экран, а
  события — из экрана во ViewModel. Состояние хранится в одном месте и не
  правят из десяти мест.

Котировки приходят по [[WebSocket и fan-out котировок|WebSocket]]. ViewModel
обновляет `MutableStateFlow`, после чего Compose перерисовывает строки. График
инструмента рисуется на `Canvas`.

DI (внедрение зависимостей) здесь **ручной** — простой `AppContainer` создаёт и
раздаёт зависимости, без Dagger/Hilt. Для приложения такого размера этого
достаточно и понятнее.
