# Jetpack Compose, MVVM и StateFlow

Android-клиент написан на Jetpack Compose. Основная идея Compose в том, что
экран описывается как функция от состояния. Изменилось состояние —
экран сам перерисовался.

## Как устроен подход

Раньше в Android часто приходилось руками менять элементы интерфейса: здесь
поменять текст, там спрятать кнопку, где-то обновить список. В Compose подход
другой: есть состояние экрана, и UI показывает именно его.

Например, если в состоянии изменилась цена акции, строка в списке обновится
после изменения состояния. Не надо отдельно искать `TextView` и менять его.

## Зачем MVVM

MVVM помогает не смешивать экран и логику. Я держу правило:

- экран только рисует данные и отправляет события;
- ViewModel хранит состояние и реагирует на действия пользователя;
- репозитории и API получают данные.

Это удобно для котировок. WebSocket приносит новые цены, ViewModel обновляет
состояние, Compose показывает новый список. Экрану не важно, откуда именно
пришла цена.

## StateFlow

StateFlow здесь нужен как поток состояния. У него всегда есть текущее значение,
и Compose может на него подписаться.

Данные идут в одну сторону: из ViewModel в экран. Обратно идут только события:
нажал кнопку, выбрал тикер, открыл график. Из-за этого проще понять, где именно
меняется состояние.

## Техническая деталь

Пример ViewModel:

```kotlin
class MarketViewModel(...) : ViewModel() {
    private val _state = MutableStateFlow(MarketState())
    val state: StateFlow<MarketState> = _state

    fun onTick(tick: Tick) {
        _state.update { it.copy(quotes = it.quotes.updatePrice(tick)) }
    }
}
```

В UI это читается так:

```kotlin
val state by viewModel.state.collectAsStateWithLifecycle()
MarketScreen(state)
```

DI в проекте ручной, через `AppContainer`. Для такого приложения это достаточно
понятно и не перегружает архитектуру.
