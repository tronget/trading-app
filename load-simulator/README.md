# Load Simulator (#5)

Нагрузочный имитатор без GUI: поднимает тысячи виртуальных клиентов против
Gateway. Каждый клиент регистрируется (при повторном запуске — логинится)
и в цикле выполняет случайные действия с весами:

| Действие | Вес | Запрос |
|---|---|---|
| quotes | 0.45 | `GET /quotes` |
| portfolio | 0.20 | `GET /portfolio` |
| order | 0.15 | `POST /orders` (market, случайный символ/сторона) |
| trades | 0.10 | `GET /trades?limit=20` |
| history | 0.10 | `GET /quotes/{symbol}/history` |

Часть клиентов (`-ws-clients`) дополнительно держит WebSocket-подписку на все
символы — проверяется стабильность стрима под нагрузкой.

## Запуск

```bash
ulimit -n 65535   # 10k соединений не влезут в лимит по умолчанию
go run . -clients 10000 -ws-clients 1000 -ramp-up 60s -duration 90s \
         -interval 5s -gateway http://localhost:8080 -output report.json
```

`-interval` — пауза между запросами одного клиента (5s × 10k клиентов ≈ 2000 RPS).
Клиенты стартуют плавно в течение `-ramp-up` и рассинхронизированы джиттером.

На стороне Gateway для теста с одного хоста нужно поднять лимит на IP:
`RATE_LIMIT_RPS=200000` (иначе все клиенты упрутся в общий лимит одного IP)
и желательно `BCRYPT_COST=6`, чтобы массовая регистрация не съела CPU.

## Результат

Таблица в консоль + JSON-отчёт: count/errors/p50/p95/p99/max по каждому
действию, число полученных тиков и сбоев WS. Ошибкой считаются сетевые сбои
и 5xx (2xx/4xx — штатные ответы).

Зафиксированный прогон на 10 000 клиентов — `../report/load/load-test-10k.json`:
255 193 запроса за 90 с (~2 800 RPS), 10 ошибок (0.004%), p99 ≤ 210 мс,
1000 WS-соединений приняли 6.08 млн тиков без переподключений.
