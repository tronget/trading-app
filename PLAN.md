# План проекта: мобильное приложение для трейдинга и инвестиций

> Учебный проект по дисциплине «Разработка мобильных приложений».
> Цель — построить полную экосистему биржевого терминала: от драйвера-имитатора
> котировок в ядре Linux до нативного Android-приложения, способную обслуживать
> **10 000 одновременных клиентов**.
>
> Этот документ — пошаговый план «от начала до конца». Он рассчитан на то, чтобы
> по нему можно было вести реализацию (в т.ч. с помощью ИИ-агента): каждый этап
> содержит конкретные задачи, контракты API, схемы данных и критерии готовности.

---

## 1. Что уже сделано

| # | Компонент | Язык | Статус | Папка |
|---|-----------|------|--------|-------|
| 7 | Драйвер Linux с имитацией котировок | C | ✅ Готово | `stock_gen_driver/` |
| 6 | Микросервис получения котировок | Go | ✅ Готово | `adapterForCdriver/` |

**Драйвер `stock_gen.c`:**
- Символьные устройства `/dev/stock0..3`, модель геометрического броуновского движения (GBM).
- Бинарная запись 32 байта (LE): `seq u64`, `timestamp_ns s64`, `price_udollar s64`, `volume u32`, `dev_idx u32`.
- `ioctl`: `SET_PARAMS` / `GET_PARAMS` / `RESET` (базовая цена, волатильность, дрифт, интервал тика).
- Блокирующее чтение + `poll`, кольцевой буфер на 64 записи на устройство.

**Go-сервис `adapterForCdriver/main.go`:**
- Читает устройства, хранит снапшоты и историю в памяти.
- REST: `GET /health`, `/v1/devices`, `/v1/ticks/latest[?device=N]`, `/v1/ticks/history?device=N&limit=K`.
- Стрим в реальном времени: `GET /v1/stream[?device=N]` (Server-Sent Events).
- Корректно стартует даже без `/dev/stock*` (режим `degraded`) — удобно для разработки на macOS.

## 3. Архитектура системы

```
┌──────────────────────────────────────────────────────────────────────────┐
│                              КЛИЕНТЫ                                        │
│   ┌────────────────────┐   ┌────────────────────┐   ┌──────────────────┐   │
│   │ Android (Kotlin +  │   │ React Native       │   │ Нагрузочный      │   │
│   │ Jetpack Compose) #1│   │ (опц.)          #2 │   │ имитатор 10k  #5 │   │
│   └─────────┬──────────┘   └─────────┬──────────┘   └────────┬─────────┘   │
└─────────────┼────────────────────────┼────────────────────────┼───────────┘
              │  REST (JSON) + WebSocket/SSE для котировок        │
              └────────────────────────┬─────────────────────────┘
                                       ▼
                       ┌───────────────────────────────┐
                       │  #3 GATEWAY (Ktor, Kotlin)      │  ← аутентификация (JWT),
                       │  - /auth, /portfolio, /orders   │     rate limiting,
                       │  - /quotes, /ws (стрим котировок)│     агрегация ответов
                       └───────┬───────────────┬─────────┘
                               │               │
              ┌────────────────┘               └──────────────────┐
              ▼                                                    ▼
   ┌──────────────────────┐                          ┌────────────────────────┐
   │ #4 DATA-СЕРВИС (Ktor) │                          │ #6 QUOTES (Go) — ГОТОВ │
   │ счета/ордера/сделки   │                          │ читает драйвер, отдаёт │
   │ матчинг, портфели     │                          │ /v1/stream (SSE)       │
   └───┬───────────┬───────┘                          └───────────┬────────────┘
       │           │                                              │
       ▼           ▼                                              ▼
┌────────────┐ ┌──────────┐   ┌──────────────┐          ┌──────────────────────┐
│ PostgreSQL │ │ Redis/   │   │ ClickHouse   │          │ #7 Драйвер C — ГОТОВ │
│ (счета,    │ │ KeyDB    │   │ (история     │          │ /dev/stock0..3       │
│  ордера)   │ │ кэш+брокер│  │  котировок,  │          └──────────────────────┘
└────────────┘ └──────────┘   │  тики, метрики)│
                              └──────────────┘
                  ┌─────────────────────────────────────────┐
                  │ OpenTelemetry Collector → трейсы/метрики  │
                  └─────────────────────────────────────────┘
```

**Поток данных «котировки» (read path):**
Драйвер C → Go-сервис (#6) → публикует тики в Redis Pub/Sub + пишет в ClickHouse → Gateway (#3) подписан на Redis → push клиентам по WebSocket/SSE.

**Поток данных «торговля» (write path):**
Клиент → Gateway (#3) `POST /orders` → Data-сервис (#4) проверяет баланс, исполняет ордер по последней цене (из Redis-кэша) → пишет сделку в PostgreSQL → обновляет портфель → ответ клиенту.

## 4. Технологический стек (из требований)

- **Android:** Kotlin, Jetpack Compose, Retrofit/Ktor-client, Coroutines + Flow, MVVM (ViewModel + StateFlow), DataStore (токен), MPAndroidChart/Compose-графики.
- **Бэкенд:** Kotlin, **Ktor**, корутины, `kotlinx.serialization`, голый SQL (без ORM — по требованию слайда), JDBC + HikariCP.
- **Сбор котировок:** Go (готово).
- **Драйвер:** C (готово).
- **Брокер/кэш:** Redis Server или KeyDB (Pub/Sub + кэш последних цен).
- **СУБД:** PostgreSQL (счета, ордера, сделки, пользователи).
- **Большие объёмы / быстрая выдача:** ClickHouse (история тиков, свечи, аналитика).
- **Логи/трейсы:** OpenTelemetry (+ коллектор, например Jaeger/Tempo для просмотра).
- **VCS:** Git.
- **Документация:** Dokka (KDoc → HTML) + Markdown + Obsidian.
- **Отчёт:** PDF по ГОСТ (оглавление, введение, заключение, список литературы, иллюстрации).

## 6. Публичный контракт API (Gateway → клиенты)

> Базовый URL: `https://<host>/`. Авторизация: `Authorization: Bearer <JWT>`.

**Аутентификация**
```
POST /auth/register   { "login": "...", "password": "..." }            → 201 { userId, accessToken, refreshToken }
POST /auth/login      { "login": "...", "password": "..." }            → 200 { accessToken, refreshToken }
POST /auth/refresh    { "refreshToken": "..." }                        → 200 { accessToken }
```

**Котировки**
```
GET  /quotes                                  → 200 [ { symbol, name, last, changePct, volume } ]
GET  /quotes/{symbol}/history?from&to&interval→ 200 [ { t, open, high, low, close, volume } ]   // свечи из ClickHouse
GET  /ws            (WebSocket)               → subscribe { "action":"subscribe","symbols":["SBER","GAZP"] }
                                                ← { symbol, price, ts, volume }                  // поток тиков
```

**Торговля и портфель**
```
GET    /portfolio                             → 200 { cash, positions:[ { symbol, qty, avgPrice, last, pnl } ], totalValue }
GET    /orders                                → 200 [ Order ]
POST   /orders   { symbol, side, type, qty, price?, clientOrderId } → 201 { orderId, status }
DELETE /orders/{orderId}                       → 200 { status: "CANCELLED" }
GET    /trades                                → 200 [ Trade ]
```

Поля: `side ∈ {BUY, SELL}`, `type ∈ {MARKET, LIMIT}`, `status ∈ {NEW, FILLED, PARTIALLY_FILLED, CANCELLED, REJECTED}`.

Индексы: `orders(status, symbol)` (для матчинга лимиток), `trades(user_id, executed_at)`.
Все изменения баланса/позиции/сделки — в одной транзакции.

### Redis / KeyDB
```

PUBLISH quotes:ticks     → JSON тика                            // брокер: Go #6 → Gateway #3, Data #4

session/refresh-токены   → опционально
```

## 10. Риски и заметки

- **macOS для разработки:** драйвер ядра — только Linux. Бэкенд разрабатывать на macOS можно (Go-сервис работает в режиме `degraded`); для реальных котировок — Linux/WSL/VM или docker-контейнер с подключённым устройством.
- **Голый SQL** (требование слайда) — без ORM; следить за SQL-инъекциями (только параметризованные запросы).
- **10k клиентов:** узкое место обычно не CPU, а лимиты сокетов/файловых дескрипторов — поднять `ulimit -n`, настроить пулы соединений и rate limiting.
- **Согласованность баланса:** все денежные операции — в транзакциях БД; идемпотентность ордеров по `clientOrderId`.
- **Матчинг лимиток:** отдельный воркер в Data-сервисе слушает `quotes:ticks` и исполняет активные заявки — не блокировать им основной HTTP-поток.
